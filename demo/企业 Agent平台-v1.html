分阶段实施路线图

我们将分三步走，确保每一步的投入都有清晰的回报，最终实现宏伟蓝图。

现状分析与核心痛点

当前状态：现有AI助手为“烟囱式”独立开发，能力局限于被动问答，知识库依赖第三方，不具备主动沟通能力。

核心痛点：重复建设成本高昂、能力无法复用、运营效率低下、服务模式单一，价值天花板低。

1

第一阶段：试点突破 (6-9个月)

聚焦“游戏运营助手”的平台化升级与能力增强

能力定位：流程化助手 (Agent v1.0)

将“手工作坊”升级为“流水线”，实现从被动应答到引导式流程化对话的跃迁。

核心任务

  • 平台架构升级：重构为模块化、可配置的SaaS服务。
  • 功能能力拓展：引入更复杂的语义理解和多轮对话管理。
  • 多游戏管理：支持快速为不同游戏部署和配置助手实例。

预期成果

  • 成功将现有助手迁移至新SaaS平台架构。
  • 新游戏接入配置时间缩短至数天内。
  • 多轮对话成功率达到70%以上。
  • 验证SaaS架构的可行性和复用价值。
2

第二阶段：通用化赋能

从“专用工具”到“通用平台”

能力定位：策略交互型助手 (Agent v2.0)

将“生产线”升级为“通用工厂”，实现从“被动”到“主动”的关键跨越，赋能新业务部门。

核心任务

  • 能力抽象与通用化：提炼核心AI能力，支持多元业务场景。
  • 扩展连接能力：对接核心办公系统，为主动服务奠基。
  • 构建模板市场：创建客服、HR助手模板，降低使用门槛。

预期成果

  • “企业级智能助手平台”1.0版上线。
  • 成功赋能2-3个新业务部门。
  • 支持他们自主配置和管理不同类型的智能助手。
3

第三阶段：生态化运营

从“中央平台”到“赋能生态”

能力定位：自主协作型智能体 (Agent v3.0)

推动平台走向“深度智能生态”，实现“人人皆可构建，处处皆有助手”的终极愿景。

核心任务

  • 全面推广应用：在HR、IT、财务、销售等更多部门落地。
  • 建立模板与组件市场：促进能力复用，降低使用门槛。
  • 持续运营迭代：赋能业务人员成为“Agent构建师”。

预期成果

  • 智能化成为公司日常运营的基石。
  • 核心指标达成或超过项目总体目标。
  • 平台成为企业AI能力和数据资产的核心中心。

首个MVP版本详细计划 (3个月)

冲锋的号角已经吹响，这是我们迈向成功的第一步。

MVP 战略定位与目标说明

本次MVP是平台宏大战略的“滩头阵地”和“最小可行闭环”。其核心价值在于：验证核心价值主张、验证核心技术架构、以及建立内外部信心。一切设计都将围绕“用最小的代价，跑通一个从‘助手配置’到‘玩家服务’再到‘数据反馈’的完整业务闭环”这一原则展开。

MVP 三大核心目标

平台可见

上线基础SaaS平台与运营工作台，支持运营人员无代码配置助手。

服务可见

实现核心“被动响应”场景，有效处理2-3个高频运营问题。

关怀可见

验证关键“主动关怀”场景，如基于事件自动发送祝贺消息。

MVP 能力边界

包含的功能/特性

  • 运营后台工作台: 核心！提供知识库管理、对话脚本管理(YAML/JSON)、主动关怀规则配置。
  • DM引擎: 支持基于已录入脚本的状态机驱动多轮对话。
  • LLM基础集成: 通过RAG框架结合游戏知识库增强问答。
  • 事件订阅接口: 提供简单API接收外部事件，支撑“主动关怀”。

“不做”或“简化”的功能

  • 可视化对话流设计器: 最主要简化项，采用脚本管理。
  • 高级运营功能: 如A/B测试、灰度发布、复杂权限管理。
  • 高级AI能力: 如情感分析、用户画像、模型微调。
  • 复杂流程自动化: 不提供可视化工作流编排引擎。

各阶段AI能力实现路径

清晰规划每个阶段AI能力的建设重点，确保技术与业务同步演进。

MVP版本 (AI平台雏形 v0.1)

Agent能力目标: 聚焦被动响应 (RAG) 和简单主动关怀 (事件驱动)。

NLU服务 (v0.1 - 场景聚焦)

技术实现: 基于 Langchain 框架开发意图识别和实体提取能力，封装成初步服务。

关键路径: 明确场景 → LLM提示词构建 → 提供简单API。

DM服务 (v0.1 - 脚本驱动)

技术实现: 构建对话脚本解释器（支持JSON/YAML），根据NLU结果加载并执行对应脚本。

关键路径: 定义脚本格式 → 开发脚本解释器 → 实现状态管理。

LLM集成服务 (v0.1 - RAG核心)

技术实现: 重点实现RAG框架，负责问题向量化、知识检索、Prompt组合、调用LLM API。

关键路径: LLM API选型与接入 → RAG流程实现 → Prompt设计。

知识管理服务 (KMS) (v0.1)

技术实现: 后端提供知识的增删改查接口，支持运营人员在后台上传和管理FAQ格式的知识，知识存储于向量数据库（用于RAG）和/或关系型数据库

关键路径: 选型向量数据库 -> 实现知识导入与向量化 -> 开发检索API。

事件订阅与简单规则引擎 (v0.1 - 主动关怀支撑)

技术实现: 提供一个API接口，供游戏后端推送特定业务事件（如玩家升级）内置一个轻量级规则匹配逻辑，根据运营后台配置的“事件-动作”规则（如“玩家升到10级 -> 发送祝贺消息”），触发DM执行相应的主动消息发送脚本

关键路径: LLM API选型与接入 → RAG流程实现 → Prompt设计。

第一阶段 (AI平台基础 v0.5)

Agent能力目标: 支持更复杂的引导式流程对话和初步个性化推荐。

NLU服务 (v0.5 - 模型优化与服务化)

技术实现: 提升意图识别和实体提取的准确率和泛化能力。将NLU彻底封装为独立的、可配置的微服务，供不同游戏实例共享和调用。

关键路径: NLU模型调优 -> 服务化改造 -> 接口标准化

DM服务 (v0.5 - 复杂逻辑支持)

技术实现: 增强对话脚本解释器，支持条件判断、循环、变量传递和“调用子脚本/子流程”机制，实现复杂分支对话。

关键路径: 扩展脚本语言功能 → 增强状态机引擎 → 完善上下文管理。

LLM集成服务 (v0.5 - Prompt工程与管理)

技术实现: 建立初步的Prompt模板库和管理机制，针对不同场景优化Prompt。探索更精细化的RAG策略（如多路召回、答案重排）

关键路径: Prompt版本控制 -> RAG策略优化

用户理解与个性化引擎 (雏形 v0.1)

技术实现: 收集用户行为数据和关键标签，基于简单用户分群和规则引擎实现个性化内容触达。

关键路径: 用户数据模型设计 → 规则引擎集成。

第二阶段 (AI平台 v1.0)

Agent能力目标: 赋能新业务部门，实现基于事件的智能主动服务。

DM服务 (v1.0 - 可视化与策略化)

技术实现: 探索提供简化版可视化对话流设计工具，并引入业务策略模型。DM能动态判断并调用子对话或触发Workflow来补全信息。

关键路径: 可视化编辑器技术选型与开发 → 策略引擎集成。

企业知识中枢 (KMS v1.0 - 知识融合)

技术实现: 支持接入更广泛的企业数据源（如钉钉文档），开始构建基础的企业知识图谱。

关键路径: 多源数据ETL → 知识图谱技术引入。

智能流程自动化引擎 (雏形 v0.5)

技术实现: 增强事件总线能力，订阅企业系统事件。支持通过简单配置，串联AI能力和API调用，实现初步的流程自动化。

关键路径: 企业事件总线集成 → 轻量级流程编排能力。
